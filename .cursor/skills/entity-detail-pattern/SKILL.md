# Skill: Entity Detail Pattern

## When to use
- Any detail page for Property / Party / Contract / ServiceRequest
- Detail page redesign or new detail routes

## Objective
Keep entity detail pages structurally consistent across modules.

## Canonical Layout Pattern
1. Header summary card
2. Main information block
3. Related records block
4. Attachments block
5. Timeline/history block
6. Right-side quick actions

## Rules
- Do not invent unique layout pattern per entity unless justified.
- Keep action placement predictable.
- Keep relationship visibility high (linked entities and outputs).

## Checklist
- Is this page immediately scannable?
- Can user find history and attachments in one pass?
- Are quick actions context-aware and traceable?
